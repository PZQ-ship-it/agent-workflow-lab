# Agent 工作流方法比较图

状态：draft
更新时间：2026-06-24

## 目的

这份笔记把第一轮检索到的材料整理成一个中文可读的比较框架。它不是为了给每个新词下最终定义，而是为了回答几个更实用的问题：

- 这个方法到底解决什么问题？
- 什么时候应该用它？
- 什么时候不值得用？
- 它通常会怎么失败？
- 第一轮该怎么做小实验验证？

## 快速判断表

| 方法 | 主要解决 | 适合使用 | 不适合使用 | 初步测试 |
|---|---|---|---|---|
| 项目规则（`AGENTS.md` / `CLAUDE.md`） | 仓库级长期约定 | 规则几乎每次在这个 repo 工作都要遵守 | 指令很长、很少用、只属于某个具体流程 | 同一个任务分别在有/无规则文件下跑一次 |
| Skill | 可复用 SOP / 能力包 | 一个流程会重复出现，且需要步骤、参考资料、模板或脚本 | 只是一次性说明，或纯项目背景 | 用 3 个 prompt 测触发准确率和输出质量 |
| Harness | 模型之外的执行环境 | 可靠性依赖工具、权限、状态、验证和可重复执行 | 简单问答或一次性文本输出已经够用 | 重跑同一任务，对比 trace、diff、测试和恢复能力 |
| Loop | 带停止条件的反复改进 | 任务需要执行 -> 验证 -> 修复 -> 汇报 | 没有退出条件，也没有可观察证据 | 人为制造失败，看 loop 是否收敛或空转 |
| Goal / 长任务状态 | 持久目标和停止标准 | 任务跨多轮、跨上下文压缩，或需要长时间运行 | 目标很泛，验收标准主观 | 中断后恢复，看状态是否足够接着做 |
| Subagent | 上下文隔离和专业分工 | 搜索、审查、领域判断会污染主上下文 | 分工没有明确输出格式 | 单 agent review vs 多角色 reviewer 对比 |
| Hook | 生命周期上的确定性守门 | 必须在 prompt、tool、stop 等边界强制触发规则 | 误拦截会严重影响正常工作 | 构造允许/拦截样例，测误报和漏报 |
| MCP / Tool server | 结构化外部能力 | agent 需要访问实时外部状态、API、应用或专门运行时 | 浏览器或 shell 已经简单可靠 | 同一查询分别走 MCP 和 fallback，对比完整性与稳定性 |
| Worktree 隔离 | 并行任务安全执行 | 多个 agent / 多个任务可能同时改代码 | 单个小改动，或不是 git repo | 两个 worktree 并行改动，检查冲突和合并成本 |
| Eval / trace loop | 可度量地改进 harness | 有重复失败案例，且能记录 trace 或评分 | 没有稳定任务集或成功指标 | 把 5 个失败变成 eval，改 harness 前后对比 |

## 各方法说明

### 项目规则

项目规则是最低成本的控制层，典型文件是 `AGENTS.md`、`CLAUDE.md`、仓库内的 project instructions。它适合放 agent 几乎每次进入这个 repo 都应该知道的东西。

适合放：

- 包管理器和测试命令；
- 架构边界和禁止改动范围；
- 安全规则；
- 本 repo 的命名、目录、验证方式；
- 修改代码后必须跑的命令。

不适合放：

- 很长的业务背景；
- 某个罕见流程的详细 SOP；
- 一次性任务说明；
- 大段 API 文档或外部文章。

常见失败：

- 把规则文件写成知识库，太长导致 agent 抓不到重点。
- 临时任务和长期规则混在一起。
- 本来应该抽成 skill 的流程，硬塞进 `AGENTS.md`。

第一轮实验：

- 选一个小 repo 任务。
- 先用极简规则跑一次，再加完整 `AGENTS.md` 跑一次。
- 对比：是否少读无关文件、是否选对命令、是否跑了验证、diff 是否更收敛。

### Skill

Skill 可以理解成“可复用的 SOP + 可选脚本/模板/参考资料”。它适合那些不是每次都要用，但一旦用到就需要一套明确流程的任务。

适合用 skill：

- 论文 digest；
- 代码审查；
- benchmark 流程；
- release checklist；
- visual QA；
- PPT / PDF / HTML 处理流程；
- 特定领域的排障流程。

不适合用 skill：

- 只是一个命令；
- 只服务于一个文件的一次性操作；
- 没有明确触发条件；
- 包含不应该跨项目传播的私密上下文。

常见失败：

- `description` 太泛，导致乱触发。
- `description` 太窄，导致该触发时不触发。
- skill 和项目规则重复，后续不同步。
- 明明自然语言步骤够用，却写了脆弱脚本。

第一轮实验：

- 把同一个流程分别写成 skill 和项目规则。
- 用 3 类 prompt 测试：显式调用、隐式相关、不相关。
- 记录触发准确率、输出稳定性和维护成本。

### Harness

Harness 是模型外面的工作环境，不是单个功能。它包括工具、权限、上下文加载、记忆、sandbox、worktree、测试、trace、状态文件、反馈回路等。好的 harness 让模型更像一个能在真实工程环境里工作的执行者。

适合用 harness 思维：

- agent 失败不是因为“没想明白”，而是缺工具、缺反馈、缺状态。
- 你反复用 prompt 修同一个问题。
- 你关心的是可重复成功，而不是某次跑通。
- 你需要审计、恢复、handoff 或长期运行。

常见失败：

- 流程还不稳定时就搭大框架。
- 本来 checklist 能解决，却上复杂 orchestration。
- raw 状态抓太多，带来隐私风险。
- 只优化 demo，不优化失败恢复。

第一轮实验：

- 看一个最小 harness 项目，比如 Pu.sh 或 Zot。
- 只复现最小闭环：读文件、改文件、跑检查、失败后恢复。
- 和普通 agent 用法在同一个 fixture 上对比。

### Loop

Loop 不是“让 agent 一直试”。一个有价值的 loop 必须有状态、证据和停止条件。

好的 loop 至少包括：

- 固定目标；
- 状态文件或进度标记；
- 验证命令或可观察信号；
- 重试预算；
- 失败升级规则；
- 最终报告格式。

常见类型：

- 实现 -> 测试 -> 修复；
- benchmark -> profile -> patch -> rerun；
- trace -> 归因失败 -> 加 eval -> 改 harness；
- 发现 -> 分流 -> 执行 -> 验证 -> 记录。

常见失败：

- 无限润色；
- 目标悄悄漂移；
- 昂贵检查反复跑，但没有新信息；
- 把 flaky 信号当成可靠事实。

第一轮实验：

- 做一个很小的失败测试 fixture。
- 让 agent loop 最多尝试 3 次。
- 看它是否记录尝试、是否按条件停止、是否解释剩余风险。

### Goal / 长任务状态

Goal 或长任务状态适合跨多轮、跨上下文压缩、跨时间的任务。它应该保存目标、约束、验收标准、证据、阻塞点和当前步骤。

适合用：

- 任务可能跑几小时；
- 有多个 milestone；
- 人可能离开后再回来；
- agent 必须知道什么时候不该继续。

常见失败：

- 目标太大太虚；
- 验收标准不可观察；
- 状态文件变成流水账，而不是操作记录；
- agent 因为时间/预算快没了就宣布完成。

第一轮实验：

- 用 `SPEC.md`、`PLAN.md`、`STATUS.md` 做一个小功能。
- 完成一个 milestone 后中断。
- 换一个上下文恢复，看是否只靠磁盘状态就能继续。

### Subagent

Subagent 最重要的价值不是并行，而是上下文隔离。它适合把探索、审查、资料收集、专业判断丢到一个干净上下文里，最后只把结论交回主 agent。

适合用在：

- 安全审查；
- 测试缺口分析；
- 代码库搜索；
- 设计批判；
- 资料收集；
- 比较多个实现方案。

主 agent 应该保留：

- 最终决策；
- 跨领域权衡；
- 面向用户的总结；
- 文件修改，除非明确授权 subagent 改。

常见失败：

- 派活太模糊，返回也很空。
- agent 太多，噪声成倍增加。
- 多个 subagent 重复搜索同一批上下文。
- 主 agent 不验证证据就接受结论。

第一轮实验：

- 同一个 diff，先让单 agent review。
- 再拆成 4 个 reviewer：正确性、安全、测试、可维护性。
- 对比独有发现、误报、耗时和成本。

### Hook

Hook 是生命周期边界上的确定性守门员。它比 prompt 里的要求更硬，因为它在模型之外执行。

适合用在：

- prompt 提交前扫描 secret；
- shell 命令前阻止危险操作；
- turn 结束后强制验证；
- 注入少量当前上下文；
- 记录工具调用审计日志。

不适合用在：

- 需要复杂人类判断的规则；
- 误报会严重打断正常工作的场景；
- 一个简单测试或命令就能解决的问题。

常见失败：

- 正则太脆，误拦正常命令。
- hook 输出污染上下文。
- 隐藏行为太多，系统不好 debug。
- stop hook 制造不想要的自动续跑。

第一轮实验：

- 写一个很小的 prompt/tool guard。
- 阻止明显 secret 和危险删除命令。
- 用 10 个 allow/block 样例测误报和漏报。

### MCP / Tool Server

MCP 或 tool server 适合给 agent 结构化外部能力，例如搜索、Figma、浏览器状态、API、数据库、本地应用或专门运行时。

适合用 MCP：

- 工具有稳定 schema；
- 比浏览器/shell 更可靠；
- 登录态能留在本地；
- 结果需要结构化字段或 provenance；
- 失败时能清楚降级。

常见失败：

- 登录态悄悄过期。
- MCP 反而不如 CLI / API 稳。
- 把工具结果当成权威，不做交叉验证。
- 凭据进入日志或仓库。

第一轮实验：

- 选一个查询任务。
- 分别用 MCP 和公开 fallback 路径跑。
- 对比完整性、速度、失败提示和引用质量。

### Worktree 隔离

Worktree 隔离是 harness 的一种执行层技术，用来避免多个 agent 或多个任务互相污染工作区。

适合用：

- 多个任务可能并行改同一仓库；
- 每个任务可以独立分支和测试；
- 失败尝试要能干净丢弃；
- 需要每个任务有清晰 diff。

常见失败：

- 共享 cache、端口、生成物仍然冲突。
- 合并成本超过隔离收益。
- agent 不小心改了父仓库状态。
- worktree 名称、任务 ID、分支名对不上。

第一轮实验：

- 建两个小 feature worktree。
- 各跑一个独立任务。
- 最后比较 diff 和冲突情况。

### Eval / Trace Loop

Eval loop 是用来改进 harness 本身的。关注点不是某一次回答，而是把重复失败变成可测样例，再比较 harness 修改前后的效果。

适合用：

- 有重复失败；
- 能捕获 trace 或样例；
- 成功标准可以脚本、rubric 或人工标注；
- 你想比较 harness 改动，而不是换模型。

常见失败：

- eval 集太小，或者只为某个 fix 量身定制。
- 指标奖励表面行为。
- trace 里混入私密内容。
- 改进过拟合简单案例。

第一轮实验：

- 从一个 workflow 里收集 5 个失败。
- 写清预期结果。
- 只改一个 harness 组件。
- 前后重跑对比。

## 当前观察到的趋势

| 趋势 | 证据来源 | 下一步测试 |
|---|---|---|
| Claude 规划，Codex 执行 | 小红书、Reddit、公众号发现 | 固定任务和验收标准，测试双 agent handoff |
| 不要裸用 agent | 知乎、小红书、官方 harness 叙事 | 对比裸 prompt、项目规则、skill/harness |
| Skill 成为可移植 SOP | OpenAI docs、Anthropic docs、知乎 AI-Infra skills | 写一个跨 repo skill，测试触发可靠性 |
| Harness 极简化 | HN 的 Pu.sh、Zot 等项目 | 复现最小 read/edit/test loop |
| 长任务监督 | OpenAI 长任务文章、小红书长任务经验 | 用 SPEC/PLAN/STATUS 测中断恢复 |
| 多 agent review | 官方 subagents、论坛案例 | 单 reviewer vs 多角色 reviewer |
| Agent 配置打包 | VAEN、plugin/skill 仓库 | 判断本仓库是否需要模板 bundle 格式 |

## 本仓库优先级

最值得先测：

1. Skill vs 项目规则 vs script-backed harness。
2. Claude planning -> Codex execution handoff。
3. 极简 harness loop。
4. Subagent review 边界。
5. 长任务 goal/state 文件模式。

暂时不优先：

- 重型社交平台爬取；
- 大规模 multi-agent team；
- 完整自定义 MCP server；
- 完全自主后台 agent。

## 工作结论

有用的比较轴不是“哪个工具品牌更强”，而是：控制权放在哪里？

- 放在长期项目记忆里：用项目规则。
- 放在可复用流程知识里：用 skill。
- 放在确定性生命周期守门里：用 hook。
- 放在外部能力访问里：用 MCP。
- 放在上下文隔离里：用 subagent。
- 放在可重复执行和验证里：用 harness。
- 放在长任务目标和停止标准里：用 goal / status 文件。

下一步应该做小实验，每次只改变一个控制层。
